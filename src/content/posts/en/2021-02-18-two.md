I released a simple web service called [two](https://two.shuymn.me).

The source code is available at [shuymn/two](https://github.com/shuymn/two).

The goal of this project is to solve a problem I was having where OGP (Open Graph Protocol) content wouldn't expand when sharing Twitter URLs in chat tools like Slack or Teams.

(Slack has a Twitter bot application that expands tweet URLs, so this is primarily targeted at Teams. Furthermore, it should work with any service—beyond Slack and Teams—that attempts to expand OGP.)

I've been using Teams more often lately, and it doesn't expand tweet URLs at all. It was difficult to understand the content of a shared URL without clicking it, opening a separate browser application, and finally determining if it was relevant to me. This raised the barrier to information sharing, so I spent my time off building this all at once.

Technically, when a Twitter URL is received, the service dynamically fetches the page's OGP. It then generates a page containing that OGP (meta tags) and includes JavaScript that immediately redirects to the original Twitter URL upon access, issuing a new URL for the user.

For a page like https://**twitter.com**/shuymn, it works by simply replacing the domain, making it accessible via https://**two.shuymn.me**/shuymn. (Note: user profile pages do not have OGP by default, so OGP will not be displayed on the converted URL either.)

Since it is a hassle to copy a Twitter URL and then visit https://two.shuymn.me, I also created a Chrome extension called [Copy as Two](https://chrome.google.com/webstore/detail/copy-as-two/cfeedmflkhahbmjpgiiepndkhpbamllc). This extension adds a right-click menu to Twitter pages that copies the converted URL for you. It’s no exaggeration to say that I built the web service specifically because I wanted to make this extension.

By the way, as a total amateur at frontend development (or so I tell myself), the implementation is quite naive and, honestly, not great. However, the tech stack uses Next.js, just like this blog. You could say that the experience I gained building this blog came in handy.

## Does Twitter's OGP not exist?

To conclude, if you open a Twitter page in a web browser like Chrome and check the HTML, OGP is indeed configured.

So why isn't it expanding? When accessing a Twitter page without providing a User Agent, the server returns HTML containing text like "This browser is not supported," and this fallback HTML does not have OGP set. (You can see this clearly if you use a command-line tool like `curl`.)

I suspect that the systems attempting to fetch OGP do not set a User Agent, which leads them to receive the response without the OGP tags, resulting in the failure to display them. (Since I don't know the exact inner workings of OGP scraping, this is just a guess.)

## Known issues

This is unavoidable, but since it's just expanding the OGP, long tweets may exceed the character limit that the OGP display can handle, so you won't be able to see the entire message.

If you want a proper solution, like the behavior of the Twitter app for Slack (in the case of Teams), there is something called Outgoing Webhooks that looks like it could solve this. However, that approach involves creating an Outgoing Webhook for every channel, and even if you get past that, it poses the risk that all channel conversations would flow to the webhook's destination—which might not satisfy information security requirements at some companies.

For that reason, while I don't think this is a perfect solution, I believe it is the most cost-effective method.
